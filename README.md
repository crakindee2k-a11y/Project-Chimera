# QueueStorm Investigator

Minimal full-stack submission for **SUST CSE Carnival 2026 · Codex Community Hackathon · Online Preliminary**.

It provides:

- `GET /health` returning `{ "status": "ok" }`
- `POST /analyze-ticket` returning the required ticket-analysis JSON contract
- Bare browser UI at `/`
- Deterministic rule-based analyzer, no external LLM/API key required

## Quick start

```bash
npm install
npm test
npm start
```

Open:

```text
http://localhost:3000/
```

## API

### Health

```bash
curl http://localhost:3000/health
```

Response:

```json
{"status":"ok"}
```

### Analyze ticket

```bash
curl -X POST http://localhost:3000/analyze-ticket \
  -H 'Content-Type: application/json' \
  -d '{
    "ticket_id":"TKT-001",
    "complaint":"Customer says payment failed but money was deducted.",
    "transaction_history":[]
  }'
```

Required input fields:

- `ticket_id`
- `complaint`

Optional input fields:

- `language`
- `channel`
- `user_type`
- `campaign_context`
- `transaction_history`
- `metadata`

Required output fields:

- `ticket_id`
- `relevant_transaction_id`
- `evidence_verdict`
- `case_type`
- `severity`
- `department`
- `agent_summary`
- `recommended_next_action`
- `customer_reply`
- `human_review_required`

Optional output fields included by this service:

- `confidence`
- `reason_codes`

### HTTP status codes

`POST /analyze-ticket` follows the problem-statement contract:

- `200` — analyzed successfully (always a schema-valid analysis JSON)
- `400` — malformed JSON body, or a required field (`ticket_id` / `complaint`) is missing or wrong-typed
- `422` — body is valid JSON with required keys present, but `complaint` is semantically empty (blank/whitespace)
- `413` — request body exceeds the `256kb` limit
- `500` — unexpected internal error (no stack traces or secrets are ever returned)

The service never crashes on malformed input; every error path returns a safe fallback JSON body.

## Tech stack

- **Runtime:** Node.js (>=18)
- **HTTP:** Express 4
- **Validation:** zod (input + output schema/enums)
- **Middleware:** cors, `express.json` (256kb limit)
- **Tests:** Node's built-in `assert` (no test framework dependency)
- **Container:** `node:20-alpine` Docker image
- **No** database, message queue, external API, or network call during analysis.

## MODELS

This service uses **no machine-learning or LLM model**. Ticket analysis is fully
**deterministic and rule-based** (regex classification, transaction scoring, evidence
reasoning, and safety sanitization in `src/analyzer.js` + `src/safety.js`).

**Why no model:**

- **Determinism & reproducibility** — identical input always yields identical output, which the judge harness can verify exactly.
- **Latency** — pure CPU rules return well within the p95 ≤ 5s target with no inference round-trip.
- **Safety** — guardrails (never request PIN/OTP/password, never promise unauthorized refunds, ignore prompt injection) are enforced by code, not by a model that could be jailbroken.
- **Cost** — **$0** inference cost. No API key, no token billing, no GPU. The container runs on CPU only.
- **Reliability** — no dependency on an external model provider's uptime or rate limits during judging.

**Cost reasoning:** total runtime cost is the hosting compute only (a single small CPU
container). There is no per-request model cost. Estimated marginal cost per request ≈ $0.

## Supported enums

From `SUST_Preli_Sample_Cases.json`:

- `evidence_verdict`: `consistent`, `inconsistent`, `insufficient_data`
- `case_type`: `wrong_transfer`, `payment_failed`, `refund_request`, `duplicate_payment`, `merchant_settlement_delay`, `agent_cash_in_issue`, `phishing_or_social_engineering`, `other`
- `severity`: `low`, `medium`, `high`, `critical`
- `department`: `customer_support`, `dispute_resolution`, `payments_ops`, `merchant_operations`, `agent_operations`, `fraud_risk`

## How analysis works

The analyzer is deterministic and rule-based:

1. Normalize complaint text and transaction history.
2. Detect case type from keywords and fraud/social-engineering patterns.
3. Extract amounts, transaction IDs, and phone hints from the complaint.
4. Score transactions by ID, amount, type, status, and counterparty match.
5. Set evidence verdict:
   - `consistent` when complaint and transaction evidence align
   - `inconsistent` when transaction history contradicts the claim
   - `insufficient_data` when no reliable supporting transaction exists
6. Route to the correct department and severity.
7. Apply safety guardrails before returning JSON.

## Safety guardrails

`customer_reply` is sanitized so it does not:

- Ask for PIN, OTP, password, CVV, or full card number
- Promise refund, reversal, account unblock, or recovery without eligibility review
- Send the customer to unofficial third parties
- Follow adversarial instructions embedded inside complaint text

Fraud, phishing, social-engineering, fake-link, OTP/PIN sharing, or impersonation complaints are forced to:

```json
{
  "case_type": "phishing_or_social_engineering",
  "department": "fraud_risk",
  "severity": "critical",
  "human_review_required": true
}
```

**Deploy / redeploy on Railway**

The service is connected to this GitHub repo, so any push to `master` triggers an
automatic Dockerfile build + deploy. Manual redeploy:

```bash
railway up                 # build + deploy current directory
railway domain             # show / create the public HTTPS URL
railway logs               # tail deploy logs
```

No secrets are required to run. `PORT` is the only environment variable (injected by the
host; defaults to `3000`). See `.env.example`.

## Assumptions and limitations

- Hidden judging can use cases beyond the sample pack, so rules are generalized rather than exact-case memorized.
- Evidence matching is best-effort from provided transaction history only.
- Human review is required for high-risk, high-value, inconsistent, ambiguous, wrong-transfer, agent, merchant-settlement, and fraud cases.
- Bengali support is keyword-based, not full language understanding.
- All transaction data used for development and testing is **synthetic** (from the public sample pack); no real customer data, PII, or secrets are stored in this repository.
